## Status

Accepted — `helm.kubebuilder.io/v1-alpha` plugin enabled in `PROJECT`; `dist/chart/` scaffolded with CRDs, RBAC (manager + per-resource admin/editor/viewer), and the metrics network policy.

## Context

We need a way to ship the operator to consumers (the homelab cluster, eventually the enterprise EKS fleet, and potentially external users via Artifact Hub). The realistic options are:

1. **kubebuilder Helm plugin** (`helm.kubebuilder.io/v1-alpha`) — generates a `dist/chart/` directory from the same kubebuilder markers that produce the CRDs and RBAC. Re-runs on `make helm`.
2. **Hand-maintained Helm chart** under `charts/renovate-operator/`, kept in sync manually with the generated manifests.
3. **Kustomize-only** — kubebuilder's default output is kustomize manifests under `config/`. Skip Helm entirely.
4. **OLM (Operator Lifecycle Manager)** — bundle as an OperatorHub-compatible package.
5. **Plain manifests** in a `release.yaml` — concatenated `config/` output, posted with each release.

The mogenius operator uses option 2 (hand-maintained chart). Their `charts/renovate-operator/crds/` directory contains a 1180-line CRD file that has to be regenerated and committed every time the Go API types change. This is exactly the drift hazard we want to avoid.

## Decision

Adopt the **`helm.kubebuilder.io/v1-alpha`** plugin to generate the Helm chart directly from kubebuilder's source of truth (the API types and RBAC markers). The chart is regenerated by `make helm`.

The chart is published to two channels:

- **OCI registry** at `ghcr.io/donaldgifford/charts/renovate-operator` for `helm install oci://...` consumption.
- **Plain HTTP repository** mirrored from the same source — only added if we discover a tooling reason to need one (e.g., older `helm` clients in CI).

Kustomize manifests under `config/` continue to be generated and remain a supported install path for users who prefer raw manifests; the Helm chart is layered on top, not a replacement.

## Consequences

### Positive

- The chart's CRDs come from the same source as the operator's API types. Schema drift between the two is structurally impossible.
- `make helm` is repeatable; CI can detect drift by running `make helm` and asserting `git diff --exit-code` on `dist/chart/`.
- Values surface (`values.yaml`) is bootstrapped by the plugin — image, tag, replicas, leader election, resource limits — so we don't write 200 lines of helpers from scratch.
- We keep the kubebuilder `config/` kustomize output as a secondary install path, which is valuable for users (and CI environments) that don't want Helm.

### Negative

- The plugin is **alpha**. Behavior may change between kubebuilder releases. We accept this and pin the kubebuilder version in our `Makefile` and `Dockerfile` build stage. Plugin upgrades are explicit, reviewed, and documented in CHANGELOG.
- The plugin's generated chart is opinionated; advanced Helm features (subcharts, library charts, `Chart.yaml` dependencies) require us to either fork the generated chart or supplement it with hand-edits in a `dist/chart/templates-extra/` overlay. We cross that bridge when we need it.

### Neutral

- We commit to regenerating the chart on every API type change. This is enforced by a `make helm-check` target run in CI.

## Alternatives Considered

### Hand-maintained Helm chart

Option 2. The mogenius approach. Predictable behavior, no plugin-stability risk, but every API type change requires a manual chart update — and as observed in the mogenius repo, this leads to chart artifacts that grow unbounded and are difficult to review. **Rejected** as drift-prone.

### Kustomize-only

Option 3. Kubebuilder's native output. Perfectly fine for sophisticated users; significantly more friction for casual consumers (homelab users, demo environments) who expect `helm install`. We keep kustomize as a secondary path but make Helm the primary delivery mechanism. **Rejected as the primary** delivery channel; **kept as a secondary**.

### OLM bundle

Option 4. Right answer if we were targeting OpenShift exclusively or wanted Artifact Hub OperatorHub listing as the primary distribution channel. Currently neither is true. Adds a non-trivial maintenance surface (bundle metadata, channel definitions, CSV file). **Rejected** as premature; revisit if/when we publish to OperatorHub.

### Plain `release.yaml`

Option 5. The lowest-friction install: `kubectl apply -f https://.../release.yaml`. We get this for free as a release artifact (kubebuilder's `make build-installer`); we publish it alongside the Helm chart. **Adopted as a tertiary** install path.
