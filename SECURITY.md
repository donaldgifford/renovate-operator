# Security Policy

## Reporting a Vulnerability

If you find a security issue in `renovate-operator`, **do not open a public
GitHub issue**. Email <dgifford06@gmail.com> with:

- A description of the issue.
- Steps to reproduce, or a minimal proof-of-concept.
- The operator version (`kubectl get deploy -n renovate-system
  renovate-operator-controller-manager -o jsonpath='{.spec.template.spec.containers[0].image}'`)
  and the Kubernetes version it was running against.

You can expect an acknowledgement within seven days. Coordinated disclosure
preferred — once a fix is ready, we'll cut a patch release on the affected
minor line(s) and credit the reporter (unless they prefer otherwise) in the
release notes.

## Security Posture

This section captures the security-relevant guarantees the operator and its
release artifacts make today. It's the "what does shipping this look like"
counterpart to the [Authorization guide](docs/usage/authorization.md), which
covers credential scopes (GitHub App permissions, Forgejo token scopes) for
operators *running* the controller.

### Supply chain

- **Container image signing.** Each release image at
  `ghcr.io/donaldgifford/renovate-operator:<tag>` is signed with
  [cosign](https://github.com/sigstore/cosign) keyless OIDC against
  Sigstore's public Fulcio CA. Verify with:

  ```bash
  cosign verify ghcr.io/donaldgifford/renovate-operator:<tag> \
    --certificate-identity-regexp 'https://github.com/donaldgifford/renovate-operator/.*' \
    --certificate-oidc-issuer https://token.actions.githubusercontent.com
  ```

- **SBOMs.** Each release artifact ships an [SPDX
  SBOM](https://spdx.dev) generated by
  [syft](https://github.com/anchore/syft) and uploaded alongside the
  release tarballs / image. SBOMs are stored as OCI artifacts on the
  same registry.

- **Helm chart.** Released as an OCI chart at
  `oci://ghcr.io/donaldgifford/charts/renovate-operator:<tag>`. Charts
  inherit the same image-signing chain since they reference the signed
  controller image.

- **Vulnerability scanning.** `make govulncheck` runs against
  `cmd/...` and `internal/...` on every CI build; release CI fails on
  any unfixed advisory.

### Pod Security

Both the **operator pod** and every **renovate worker pod** comply with
the [PodSecurity `restricted`
profile](https://kubernetes.io/docs/concepts/security/pod-security-standards/#restricted)
out of the box. They set:

- Pod-level: `runAsNonRoot=true`, `seccompProfile.type=RuntimeDefault`.
- Container-level: `allowPrivilegeEscalation=false`,
  `capabilities.drop=["ALL"]`.

This means you can label the operator's release namespace and any
`RenovateScan` namespaces with
`pod-security.kubernetes.io/enforce=restricted` and the operator's pods
will admit cleanly.

`readOnlyRootFilesystem=true` is **not** set on worker pods today —
Renovate writes scratch and config-cache files to its working tree, and
flipping the field on without an ephemeral volume would break the
worker. Tracked as a future enhancement (likely involves an `emptyDir`
mount at Renovate's cache path); not on the v0.1.x roadmap.

### Credential handling

- **Operator-namespace boundary.** Platform credentials (GitHub App
  private key, Forgejo token) live as `Secret`s in the operator's
  release namespace and are referenced through
  `RenovatePlatform.spec.auth.*.privateKeyRef` / `tokenRef`. The
  operator's `ClusterRole` only reads `Secret`s in its own namespace —
  it cannot enumerate or read Secrets in tenant namespaces.

- **Per-Run credential mirror.** When a `RenovateRun` materializes, the
  controller mirrors the credential `Secret` into the Run's namespace
  with an owner reference back to the Run. The mirror is GC'd
  automatically when the Run is deleted (TTL via Kubernetes' garbage
  collector). This keeps the worker pod from needing cross-namespace
  Secret read permissions.

- **No plaintext credentials in logs or status.** The operator
  redacts private-key references in `Status.Conditions` messages and
  in structured-logging output. Conditions surface
  `reason=SecretNotFound` / `SecretInvalid` without echoing the
  contents.

- **Renovate handles its own JWT minting.** The operator does not see or
  cache Git platform tokens beyond the initial App-private-key /
  token credential — Renovate inside the worker pod mints its own
  installation tokens from the App private key per Run.

### RBAC scope

- The operator's `ClusterRole` is named after the kubebuilder
  `projectName` (`renovate-operator-manager-role`) and grants:
  - Full CRUD on `renovate.fartlab.dev` resources cluster-wide.
  - Full CRUD on `Job` / `ConfigMap` cluster-wide (worker scheduling +
    shard config).
  - `get` on `Secret` only in the operator's release namespace (via
    namespaced `RoleBinding`, not the cluster role).
  - `create` on `tokenreviews.authentication.k8s.io` and
    `subjectaccessreviews.authorization.k8s.io` (used by the metrics
    endpoint's auth filter — only when `metrics.enabled=true`).
- The operator does **not** require any cluster-admin-equivalent
  verbs (e.g., `*` on `*` or RBAC management).

### Metrics endpoint

- Exposed on `:8443` over HTTPS with controller-runtime's
  `WithAuthenticationAndAuthorization` filter. Callers must present a
  bearer token whose `tokenreview` resolves to a subject with `get` on
  the cluster-scoped `nonResourceURL` `/metrics`. The chart bundles a
  `metrics-reader` `ClusterRole` for that purpose.
- Disable entirely by setting `metrics.enabled=false` (then neither
  service nor RBAC renders).
